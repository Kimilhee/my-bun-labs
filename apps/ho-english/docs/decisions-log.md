> ADR보다 가벼운 한 파일. 결정 시점 + 결정 + **왜**(짧게) + (필요 시) 재고 조건.
> 형식이 길어지기 시작하면 `adr/NNNN-*.md`로 승격.

## 2026-05-02 · 플랫폼: PWA(웹앱) 우선

**결정**: 1차 출시는 PWA(모바일 브라우저 + 카메라 API). 검증 후 Expo로 네이티브 이식 검토.

**왜**: `getUserMedia`/file capture로 카메라 접근 가능, App Store 심사·계정 없이 1주 내 데모 가능. 네이티브는 빌드·배포 셋업이 +3~5일.

**재고 조건**: 카메라 UX 한계가 명확해지거나(예: 노출 보정 필요), 오프라인 사용 비중이 커지면.

## 2026-05-02 · 1차 도메인: 내신(학교 교과서·부교재)

**결정**: 수능형·일반 영어 대신 내신.

**왜**: CEO 선택. 시장 폭은 좁지만 어휘·문법 모드의 가치가 더 크고, 사용자가 자기 책 사진 찍는 사적 이용 패턴이 명확.

**함의**: 출판사·학년 분포가 넓다 → 프롬프트는 "한국 고등학교 영어 교과서·부교재 수준" 으로 일반화. 끊어읽기 청크 단위는 "한 호흡으로 읽기 좋은 의미 단위" 정도로 느슨하게.

## 2026-05-02 · 첫 모드: 끊어읽기 (가설)

**결정**: walking skeleton에 들어가는 첫 모드는 끊어읽기.

**왜**: 출력 스키마 단순(`chunks[]`), 시각 효과 즉각, 학생 가치 직관적, 파이프라인 검증용 테스트베드로 적합.

**중요**: 4개 모드(끊어읽기/지문분석/어휘/문법)는 *가설*. 학생 인터뷰 5명 후 우선순위·추가·제거 결정. 끊어읽기 자체도 인터뷰로 검증 대상. (`modes/_hypotheses.md`)

## 2026-05-03 · LLM provider: Gemini API

**결정**: Anthropic 대신 Gemini API. 모델 ID `gemini-3-flash-preview` (CEO 지정).

**왜**: Anthropic API 가격이 데모 단계에 부담. Gemini 무료 tier가 후함.

**제약**: 모델 ID는 CEO가 직접 지정 — agent가 임의로 다른 모델로 바꾸지 말 것. 변경 시 CEO 확인 필수. preview 모델이라 응답 포맷·rate limit 변동 가능 → `server/index.ts`의 `MODEL_ID` 상수 한 곳에서만 관리.

**재고 조건**: preview 모델이 GA 모델로 승격, 또는 응답 품질·비용 사유로 multi-provider 필요 시 BFF 안 provider 추상화.

## 2026-05-03 · 메타앱 먼저, 본 앱 v0는 그 위에

**결정**: 본 앱(AI 자동 끊어읽기)을 짓기 전에, 사람이 손으로 끊어읽기를 라벨링하는 메타앱(`pages/label/`)을 먼저 만든다.

**왜**: (1) 본 앱이 그대로 import할 컴포넌트(`ChunkedText`, `tokenize`, BFF, 프롬프트 파이프)를 깐다. (2) 골든샘플 데이터 플라이휠을 시작한다 (사용자 늘면 데이터도 큰다). (3) "모드 정의서" 추상 문서 대신 실제 UX(터치로 슬래시 위치)가 정의가 됨. (4) OCR 단독 동작이 본 앱 분석보다 먼저 검증된다.

**제약**: 메타앱은 본 앱 v0 출시 전까지 제품의 1차 페이스. 학생용 UI는 §F1에서 추가.

---

## 2026-05-03 · OCR과 분석 분리하지 않음

**결정**: 별도 OCR API(Google Vision, AWS Textract 등) 없이 Gemini Vision에 사진 그대로 보내서 OCR + 분석을 한 호출에.

**왜**: 의존성·실패점 1단계 줄어듦. Gemini는 vision OCR 충분히 잘함. 모드별로 "OCR 결과 + 모드 시스템 인스트럭션" 한 번에 처리 가능.

**재고 조건**: OCR 정확도가 분석 모드 품질의 병목이 되면 분리 검토.

---

## 2026-05-03 · 프롬프트는 마크다운 파일로

**결정**: 모드별 system prompt를 코드 안 string literal이 아니라 `server/prompts/*.md`로 분리, 런타임에 `Bun.file().text()`로 로드.

**왜**: 비개발자(CEO)가 코드 모르고 프롬프트만 수정 가능. agent에게 "프롬프트는 여기"라는 단일 출처 지정. diff에서 프롬프트 변경이 바로 보임.

---

## 2026-05-03 · 모드 = 프롬프트 스위치

**결정**: 4개(또는 N개) 모드는 동일 입력(이미지 또는 추출 텍스트) + 다른 system instruction + 다른 JSON 출력 스키마. 모드별 별도 서비스/모듈/엔드포인트 안 만듦. BFF 엔드포인트는 `/api/analyze` 1개에서 `mode` 파라미터로 분기.

**왜**: 코드 분기 최소화. 새 모드 추가가 프롬프트 + 스키마 추가 = 작업 30분~수 시간. 메타앱과 본 앱이 같은 출력 스키마 공유.

---

## 2026-05-03 · UI 라이브러리: shadcn/ui는 첫 컴포넌트 필요해질 때

**결정**: 지금부터 shadcn/ui 설치하지 않음. Tailwind native 마크업으로 시작, 복합 컴포넌트(다이얼로그·드롭다운 등) 첫 등장 시 그때 설치.

**왜**: 메타앱은 form 요소 + 텍스트 렌더만 — shadcn/ui 가치 적음. 미리 설치하면 컴포넌트 생성 보일러플레이트만 늘어남.

## 2026-05-03 · 학생 인터뷰 5명을 코딩 사이에 끼움

**결정**: 본 앱 v0 walking skeleton(§F1) 진행 전에 CEO가 학생 5명 인터뷰. "마지막에 영어 지문 공부할 때 어디서 막혔어?" 같은 회고형 질문(Mom Test).

**왜**: 4개 모드는 솔루션 가설일 뿐, 학생의 진짜 잡(JTBD)은 다를 수 있음. 인터뷰 전에 첫 모드 코드 더 짜면 sunk cost 누적.

**무관 작업은 계속**: PWA 골격, BFF 파이프라인, 메타앱 라벨링은 모드와 무관 → 인터뷰 동안 병행 가능.
