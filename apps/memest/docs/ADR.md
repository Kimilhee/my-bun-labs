# memest ADR — 아키텍처 결정 기록

> 각 결정의 "왜"를 남긴다. 뒤집을 때는 해당 항목에 취소선 대신 후속 ADR을 추가할 것.
> 배경은 [PRD.md](PRD.md), 구현 상세는 [design.md](design.md).

## ADR-1. 복습 전용, 신규 암송 제외

처음 암기와 복습은 완전히 다른 처리가 필요하다는 사용자 판단. 모든 카드는 "한 번 외웠던 것"이라는 전제가 서면서 첫 만남을 진단으로 쓸 수 있게 됨(ADR-4). BTT(타자연습 프로그램) 포팅도 범위 밖 — BTT는 데이터 원천으로만 사용.

## ADR-2. 인출 사다리 + 힌트 겹 — 실패 상태의 구조적 제거

기존 카드 방식의 문제는 처음부터 자유 회상(가장 어려운 인출)을 요구해 "모르겠다"가 반복되는 것. 인식→단서 회상→자유 회상 사다리(A~D)로 단계화하고, 한 카드 안에서는 못 떠올릴수록 힌트를 한 겹씩 벗긴다(초성→첫 어절→정답). **맞춰야 다음 힌트가 아니라, 못 떠올릴수록 강한 힌트** — 방향을 혼동하지 말 것. 다음 복습은 항상 맨몸(단서만)부터: 힌트는 제시용이 아니라 측정용.

- 제목은 힌트가 아니라 **승급 축**: C = 장절+제목 제시, D = 장절만(제목까지 회상). D가 실제 암송과 동일한 조건 = 목표 상태.
- 1차 구현은 C/D만. A(노출)/B(짝짓기)는 심하게 끊어진 카드의 재활 코스로 2차.

## ADR-3. 힌트 사용량 = 채점 점수 (이진 채점 폐지)

"맞았다/틀렸다" 대신 "몇 겹의 힌트로 열렸나"라는 연속 점수를 얻는다. 승급/강등, 복습 간격, 진행도 시각화가 전부 이 값으로 돌아가고, 자가신고는 최소화된다.

## ADR-4. 단순 Leitner + 첫 만남 = 진단 배치 (FSRS 배제)

한 번 외운 내용의 재연결이라 망각곡선이 완만 → 근사 스케줄링으로 충분. 박스 0~5, 간격 [1,2,4,8,16,32]일. 별도 진단 테스트 없이 미진단 카드를 C 형태로 처음 제시하고 결과(힌트 0→D box3 / 1→C box2 / 2+→C box1 / 공개→C box0)로 배치한다. 잘 되는 절반이 노출 단계를 기어오르는 낭비를 제거.

## ADR-5. 집중 세션: 재큐잉 완주 + 비대칭 반영

사용자 요구 "한 번에 200구절, 답변 못하는 구절이 없게". 힌트를 쓴 카드는 큐 뒤로 재삽입해 맨몸으로 열릴 때까지 반복 → 종료 조건이 "완주"라서 경험상 실패가 없다. 채점은 첫 회차만 반영. **성공 신호는 간격을 연장하지 않고 약함 신호만 강등에 반영** — 몰아치기 직후의 단기기억 부풀림으로 스페이싱이 망가지는 것을 방지. 장기 간격을 늘리는 건 일일 세션의 성공만.

## ADR-6. 카드 유일성 = (파트, 제목, 장절), dedupe 금지

장절이 같아도 파트가 다르면 제목이 다르고 제목까지 암송 대상이므로 별개 카드. 파트는 시리즈 1~5, DEP 1~8이 각각 별도(총 16개). 학습 진행도도 카드별로 독립.

## ADR-7. 데이터 파이프라인: BTT 원본 → 변환기 → 정적 JSON

EUC-KR 탭 구분 원본(BTT/data/card*.txt)을 `scripts/convert-btt.ts`로 `data/verses.json`(495개)·`data/parts.json`(카탈로그)에 정규화. 원본은 수정하지 않고 변환기를 재실행 가능하게 유지.

- BTT의 AHK 파서와 동일 규칙: `@&` = 필드 내 줄바꿈(→ `note` 보존), DEP는 말씀에 탭이 섞여 3·4열 병합, 카테고리는 5열.
- DEP 대제목 8개는 원본에 없어 중제목 경계로 복원 (합계 242로 검증).
- `*` 표시 → `starred` 필드 (잘 안 외워졌던 카드 87개).
- 앱 입력 UI 대신 시드 내장: 복습 전용이라 카드 추가가 드묾.

## ADR-8. 백엔드 없음, localStorage 단일 저장소

사용자 1명, 데이터 수백 줄 — 서버는 over-engineering. 진행도·설정·진행 중 세션까지 localStorage에 저장해 앱 재실행 시 하던 카드가 그대로 복원된다. 백업은 JSON 내보내기/가져오기. localStorage는 도메인별이므로 배포 주소 변경 시 수동 이전 필요.

## ADR-9. 음성 채점: Web Speech API + "인식이 아니라 채점을 관대하게"

- STT는 브라우저 내장 Web Speech API(ko-KR). 외부 STT(Whisper, Google/Clova)는 비용·API 키·백엔드가 필요해 배제. 내장 API는 어휘 등록 불가(SpeechGrammarList는 죽은 스펙).
- STT가 옛말을 현대어로 "교정"하는 오인식(미쁘사→예쁘시고, 깨끗케→깨끗하게)은 엔진에서 못 고침 → **정답을 이미 아는 검증 문제이므로 채점 쪽을 관대하게** 하는 것이 원칙.
- **어절 단위 + 자모 분해 LCS**: 발음 유사 오기는 자모 레벨에서 대부분 흡수. 음절 단위 비교는 "미쁘사vs미프사"를 과하게 깎아 폐기.
- **옛말 사전**(`src/lib/archaic.ts`): 명시 단어 + 옛말투 어미 패턴(뇨/니라/도다/하사/케/코…). 옛말 어절은 50% 통과, 일반 80%. 발견될 때마다 사전에 추가하는 운영 방식.
- **완벽(자동 진행) 조건은 옛말 75% 이상**: 긴 옛말 어미(느니라)만 겹치고 어간이 다른 진짜 오류(이기느니라→사랑하느니라)가 50% 룰에 가려지는 것을 방지. 50~75%는 통과하되 대조 화면을 거침.
- 표시 일치율은 어절 통과율이 아니라 **실제 자모 일치율** (옛말 통과를 100%로 부풀리지 않기).
- 대조 화면은 양방향 하이라이트 + 옛말 통과 어절은 본문 단어로 교정 표시(초록).
- 오프라인에서는 STT 불가(서버 기반) → 자가 확인 모드 폴백. 온디바이스 STT는 도입하지 않음.

## ADR-10. 안드로이드 Chrome STT 우회

안드로이드는 (1) `resultIndex`를 항상 0으로 주며 전체를 반복 전송, (2) 진행 중 누적 스냅샷을 각각 새 결과 항목으로 추가("그의→그의 아들→그의 아들 안에"), (3) 짧은 침묵에 조기 종료. 대응:

- 매 이벤트 results를 0부터 재조립 (누적 금지)
- 항목이 직전 항목의 확장이면 교체, 축소면 무시하는 **스냅샷 병합**
- stop() 전 조기 종료 시 세그먼트를 저장하고 **자동 재시작**
- 카드 전환 시 cancel()로 채점 없이 폐기

## ADR-11. 배포: GitHub Pages(레포 public) + PWA 프리캐시

- dev는 `--host` + `@vitejs/plugin-basic-ssl`(마이크는 보안 컨텍스트 필수).
- 배포는 GitHub Actions → Pages. 무료 플랜은 private 레포에 Pages 불가 → 사용자가 모노레포를 public으로 전환(2026-07-14). Pages 최초 활성화는 워크플로 토큰 권한 밖이라 사용자 토큰으로 1회 수행.
- `vite-plugin-pwa`(autoUpdate)로 전체 자산 프리캐시 → 첫 방문 후 오프라인 동작. dev용 수동 서비스워커는 캐시 없이 두었다가 플러그인으로 대체.
- base는 프로덕션에서 `/my-bun-labs/`.
- **버전 규칙**: 배포되는 변경마다 package.json version을 올리고(패치=수정, 마이너=기능) 설정 하단에 표시 — 사용자가 폰에서 반영 여부를 확인하는 수단.

## ADR-12. UI 원칙: 단일 화면 + 시트

탭바·네비게이션 없음. 세션 화면 하나에 설정/구절 리스트/범위 선택이 시트로 얹힌다. 근거: 단일 목적 앱에서 "이동"은 부담.

- 앱 실행 = 하던 세션 복원. 세션 없을 때만 시작 화면.
- 본문 영역만 스크롤, 하단 버튼은 위치 고정. 버튼은 엄지 존(하단에서 버튼 높이 2배 위). 하단 스택: 보조 링크 → 큰 버튼 2개 → 지나온 구절 핸들(맨 아래).
- **스포일러 방지**: 리스트에서 현재 카드와 이번 세션 대기 카드는 장절만. 이번 세션 확인분 + 이전 세션에서 다룬 카드는 공개(전문 펼침, ⟲ 다시 암송).
- 암송 버튼은 홀드가 아니라 **토글** (꾹 누르기는 긴 구절에서 불편 — 사용자 피드백으로 변경).
- 범위 선택 시트는 즉시 적용이 아니라 **[취소]/[적용]** (모두 해제가 세션을 즉시 죽이는 사고 방지).
- 진행 중 일일 세션에서 범위를 적용하면 즉시 재구성, 집중 세션은 유지.

## ADR-13. 모노레포 디폴트 스택 준수, shadcn/ui 미도입

React 19 + Vite 6 + TS strict + Biome + Bun (루트 CLAUDE.md 디폴트). shadcn/ui는 화면 3개짜리 프로토타입에 과해서 순수 CSS 단일 파일로 시작 — 화면이 늘어나면 재검토.

## ADR-14. 대조 채점 폐기 → 따라 열림(follow-along)으로 전환 (v0.1.0)

v0.0.x의 "받아적기 대조 채점"(어절 80%/옛말 50%/완벽 75%, 양방향 하이라이트, 옛말 사전 교정 표시 — ADR-9)은 실사용 결과 **STT 정확도가 낮아 대조 자체가 신뢰를 잃음** → 폐기. 대신:

- 암송하는 동안 **본문이 말한 만큼 실시간으로 열린다.** 인식 텍스트는 표시하지 않음.
- 도달 판정은 어절당 자모 LCS **20%** 단일 임계값 — 인식이 엉망이어도 진도만 따라오면 됨.
- 막히면 본문 두 번 터치로 한 어절씩 수동 공개(자동 넘김 차단 신호).
- 완주(전부 열림)+수동 0회 = 성공 자동 진행. 그 외 결과 화면에서 자가 판단([틀렸음]/[맞은 걸로]).

이로써 옛말 사전(archaic.ts)과 세분화된 임계값들은 제거 — 정밀 채점을 포기하고 "본문을 따라 여는 보조자 + 최종 판단은 사용자"로 역할을 좁힌 것. ADR-9의 관대한 채점 원칙의 극단형이며, ADR-9~10의 자모 LCS·안드로이드 우회는 그대로 재사용.

## ADR-15. 음성인식 전면 제거 (v0.2.0)

따라 열림(ADR-14)조차 실기기(안드로이드)에서 STT 정확도·안정성이 만족스럽지 않아 **음성 기능을 전부 제거**. Web Speech API 래퍼(speech.ts), 자모 LCS 매칭(followProgress), 관련 설정(음성 암송·자동 넘김·첫소절 매칭)과 dev HTTPS(basic-ssl — 마이크 전용이었음)를 모두 걷어냄.

남은 것: **더블탭 = 다음 어절 열기** (수동 보조) + [말씀 확인] 자가 판정. 시행착오의 교훈: 브라우저 STT는 옛말 성경 본문 검증용으로는 신뢰 수준에 못 미친다. 재도입하려면 온디바이스 모델(Whisper 등) 수준의 정확도가 전제되어야 함.
